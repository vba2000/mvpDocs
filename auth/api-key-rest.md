# API key для REST

## Назначение

REST-аутентификация для внешней интеграции строится на связке:

- `API key` как публичный идентификатор ключа;
- `secret` как материал для HMAC-подписи;
- `timestamp` как защита от replay;
- опциональный IP whitelist, если он задан при выпуске ключа.

Этот способ нужен для серверных интеграций, торговых ботов, middle-layer сервисов и бэкендов партнеров.

## Базовые заголовки

| Заголовок | Обязателен | Назначение |
|-----------|------------|------------|
| `X-API-Key` | да | Публичный идентификатор ключа |
| `X-Timestamp` | да | Unix time в миллисекундах |
| `X-Signature` | да | Base64-строка HMAC-SHA256 |
| `Content-Type: application/json` | для POST | Формат тела запроса |

## Формула подписи

```text
message = "{timestamp}{METHOD}{pathWithQuery}{body}"
signature = Base64(HMAC_SHA256(secret, UTF8(message)))
```

### Что входит в `pathWithQuery`

В подпись должен попасть путь после хоста, включая query string.

Примеры:

- `GET https://cex-test.web3tech.ru/api/v1/gateway/market/time`
  -> путь для подписи: `/api/v1/gateway/market/time`
- `GET https://cex-test.web3tech.ru/api/v1/gateway/market/symbols?symbol=BTC%2FUSDT`
  -> путь для подписи: `/api/v1/gateway/market/symbols?symbol=BTC%2FUSDT`

### Что входит в `body`

- Для `GET` и `DELETE` без тела: пустая строка.
- Для `POST`: точная строка JSON, которая реально отправляется в HTTP body.

Если сериализация JSON между этапом подписи и этапом отправки отличается хотя бы пробелами, подпись станет невалидной.

## Пошаговый flow

1. Клиент формирует `timestamp` в миллисекундах.
2. Клиент собирает `pathWithQuery` ровно в том виде, как он уйдет в запрос.
3. Клиент сериализует тело запроса в строку.
4. Клиент вычисляет HMAC-SHA256 по формуле выше.
5. Клиент отправляет запрос с тремя заголовками.
6. Gateway проверяет:
   - существует ли ключ;
   - активен ли он;
   - не истек ли срок действия;
   - попадает ли IP в whitelist, если он настроен;
   - укладывается ли timestamp в допустимое окно;
   - совпадает ли HMAC-подпись.

## Ошибки авторизации

Типичный результат ошибки авторизации:

- HTTP `401`;
- обертка ответа Gateway с описанием ошибки;
- детали могут приходить в `extInfo`.

Причины:

- неверный `secret` или ошибка в строке подписи;
- неверный путь в подписи;
- подпись посчитана по другому body;
- устаревший `timestamp`;
- ключ отозван или истек;
- клиентский IP не входит в whitelist.

## Публичные и приватные маршруты

### Публичные

Методы `/market/*` не требуют `API key`. На них можно работать без подписи.

### Приватные

Для внешней интеграции по API key документируются:

- `spot`;
- `accounts` кроме операторского депозита;
- `rfq` taker-side;
- `aggregate-price`.

## Управление API keys

Выпуск и отзыв API key происходят через сессионные методы пользователя:

- `POST /auth/api-keys`
- `GET /auth/api-keys`
- `GET /auth/api-keys/{apiKeyId}`
- `DELETE /auth/api-keys/{apiKeyId}`

Эти методы нужны для жизненного цикла ключей, но не являются частью HMAC-интеграции торгового контура. Для них требуется пользовательская сессия, а не API key.

### Параметры при создании ключа

`GatewayCreateApiKeyRequest`:

| Поле | Тип | Назначение |
|------|-----|------------|
| `name` | string | Человеко-читаемое имя ключа |
| `permissions` | string[] | Набор прав: `READ`, `TRADE`, `WITHDRAW` |
| `ipWhitelist` | string[] | Разрешенные IP адреса или CIDR |
| `expiresAt` | string(date-time) | Момент истечения ключа |

### Что сохранить после выпуска ключа

При создании сервер возвращает `apiKey` и `secret`. `secret` показывается только в момент выпуска и должен быть сохранен сразу.

## Практические рекомендации

- Синхронизируйте время сервера клиента через `GET /market/time`.
- Не подписывайте уже распарсенный объект; подписывайте итоговую JSON-строку.
- Используйте отдельные ключи для разных сервисов и окружений.
- Для торговых операций всегда задавайте `clientOrderId`.
- Для account transfers задавайте `idempotencyKey`.
