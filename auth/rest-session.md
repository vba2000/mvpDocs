# REST: сессия (cookie)

## Назначение

Типичный сценарий **BFF**: пользователь логинится через OAuth2 PKCE, Gateway выставляет cookie **`GATEWAY_SESSION`**, последующие запросы автоматически привязаны к сессии.

## Как работает

1. Клиент вызывает публичный `POST /auth/login` с телом, инициирующим PKCE (см. [public/rest/auth-endpoints.md](../public/rest/auth-endpoints.md)).
2. После редиректа и callback в cookie оказывается `GATEWAY_SESSION`.
3. Фильтр сессии поднимает контекст безопасности Spring по сохранённым на Gateway токенам.

## Когда использовать сессию vs API-ключ

| Сценарий | Рекомендация |
|----------|----------------|
| Браузерное приложение на том же домене / через BFF | Сессия |
| Бот, серверная интеграция, CI | API-ключ [rest-api-key.md](rest-api-key.md) |

## Ограничения

- Для части маршрутов действует фильтр «смена пароля обязательна»: пользователи в статусе черновика/смены пароля получают ограниченный набор разрешённых путей до смены пароля. **API-ключ** в этом фильтре обрабатывается иначе (блокировка смены пароля на ключ не распространяется).
